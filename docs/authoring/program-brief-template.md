# Program Brief Template

Use this file when you want to define a new Cornerstone program before generating any YAML or markdown resources.

It is the cleanest authoring input to give a human collaborator or an AI agent.

Recommended pairing:

- fill this template first
- then attach [Program authoring brief](./track-authoring-brief.md)
- optionally attach [Simple content model](./simple-content-model.md) if the agent needs more context

## Why This Exists

Many modeling mistakes happen because the author jumps straight into checkpoints, skills, or worksheets before the learning problem is stated clearly.

The program brief prevents that.

It forces clarity on:

- who the learner is
- what the real outcome is
- how much time is available
- what constraints matter
- what should explicitly stay out of scope

## Fillable Template

Copy this section and fill it in.

```text
# Program Brief

- domain:
- audience:
- current baseline:
- target outcome:
- time horizon:
- session cadence:

- delivery mode:
- constraints:
- non-goals:
- success signals:
- tone and coaching stance:
- notes:
```

## Guidance On Each Field

- domain: maths, reading, pronunciation, music, basketball, and so on
- audience: the learner segment, not one private individual unless you explicitly want a one-off private program
- current baseline: what the learner can already do and where the weakness is
- target outcome: the real end state you want, in plain English
- time horizon: how long this slice should run before review
- session cadence: daily, three times a week, weekends only, and so on
- typical session length: keep this realistic
- delivery mode: parent-led, self-practice, coach-led, mixed, screen-first, printable-first
- constraints: attention span, printing limits, no audio, limited supervision, school alignment, and similar real constraints
- non-goals: what should stay out of scope for this slice
- success signals: what would make you say this program is working
- tone and coaching stance: direct, calm, playful, strict, adult-professional, and so on
- notes: anything else the author or agent should know

## Example 1: Age-10 Arithmetic Foundations

```text
# Program Brief

- domain: maths
- audience: age-10 learner who is weak on arithmetic fundamentals
- current baseline: knows some facts but is slow and unreliable, especially in multiplication and division; written methods are inconsistent
- target outcome: dependable fact fluency to 10 and confident written whole-number operations in short parent-led practice sessions
- time horizon: 6 weeks, reviewed every 2 weeks
- session cadence: daily
- typical session length: 15 minutes
- delivery mode: parent-led at home with printable worksheets and short oral checks
- constraints: attention span is limited; material must be calm and direct; no heavy game mechanics needed
- non-goals: fractions, decimals, algebra, and long explanations of theory
- success signals: faster recall, fewer hesitation points, correct setup of written methods, less parent prompting needed
- tone and coaching stance: plain, calm, direct, practice-first
- notes: start small and operational; split skills only when weak spots change what should be assigned next
```

## Example 2: Early Reading Starter

```text
# Program Brief

- domain: English reading
- audience: age-7 learner who can recognize many letters but is still hesitant in read-aloud practice
- current baseline: can decode some simple words but pauses often and loses confidence on short passages
- target outcome: more confident early decoding and short read-aloud fluency through parent-led daily practice
- time horizon: 4 weeks
- session cadence: daily
- typical session length: 10 to 12 minutes
- delivery mode: parent-led, printable-first, with short passages and simple prompts
- constraints: keep pages visually light and avoid long blocks of text
- non-goals: grammar drills, writing composition, and advanced comprehension work
- success signals: smoother read-aloud, fewer pauses, better decoding of simple word families
- tone and coaching stance: warm but direct, not sugary
- notes: keep checkpoint decisions simple enough for a parent to make honestly
```

## Recommended Authoring Sequence

After filling the brief:

1. ask for the program shape first
2. agree the checkpoints and skill boundaries
3. generate resources
4. generate playlists
5. review ids and references before committing anything

That order prevents premature over-modeling.
